## cosmix-mcp: lazy hub connection

- Converted `hub` field from `Arc<HubClient>` to `OnceCell<Arc<HubClient>>` — MCP server starts immediately, connects to hub on first tool call
- Eliminates startup timeout when Claude registers the MCP server but hub isn't running yet
- Added `#[tool_handler]` macro and explicit `ServerCapabilities::builder().enable_tools().build()` for proper MCP tool advertisement

## AMP CLI tooling

- Discovered websocat framing issues when talking to cosmix-hubd:
  - `--no-line` required to send multi-line AMP messages as a single WebSocket frame (default splits on newlines)
  - `-n` / `--no-close` required so websocat waits for the response instead of closing on stdin EOF
- Created `wsamp` alias: `websocat -1 -n --no-line ws://127.0.0.1:4200/ws`
- Created `amp` bash function for ergonomic AMP calls with auto-generated headers:
  - `amp mon.status` — local service call
  - `amp mon.status mko` — mesh call to remote node via `to: mon.mko.amp` addressing

## AMP protocol notes

- Routing uses `to:` header, not command prefix — `to: mon` routes to mon service, `command: mon.status` is the method
- Responses require `id:` header for hub to route reply back to caller via `pending_responses` map
- `from:` header needed so the service can address the response back
- Mesh addressing: `service.node.amp` (e.g. `mon.mko.amp`) triggers hub mesh bridge
