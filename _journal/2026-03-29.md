# 2026-03-29 — Lazy hub in cosmix-mcp, AMP CLI tooling, mesh validation

## cosmix-mcp: lazy hub connection

- Converted `hub` field from `Arc<HubClient>` to `OnceCell<Arc<HubClient>>` — MCP server starts immediately, connects to hub on first tool call
- Eliminates startup timeout when Claude registers the MCP server but hub isn't running yet
- Added `#[tool_handler]` macro and explicit `ServerCapabilities::builder().enable_tools().build()` for proper MCP tool advertisement

## AMP CLI tooling

- Discovered websocat framing issues when talking to cosmix-hubd:
  - `--no-line` required to send multi-line AMP messages as a single WebSocket frame (default splits on newlines)
  - `-n` / `--no-close` required so websocat waits for the response instead of closing on stdin EOF
- Created `wsamp` alias: `websocat -1 -n --no-line ws://127.0.0.1:4200/ws`
- Created `amp` bash function for ergonomic AMP calls with auto-generated headers:
  - `amp mon.status` — local service call
  - `amp mon.status mko` — mesh call to remote node via `to: mon.mko.amp` addressing

## AMP protocol notes

- Routing uses `to:` header, not command prefix — `to: mon` routes to mon service, `command: mon.status` is the method
- Responses require `id:` header for hub to route reply back to caller via `pending_responses` map
- `from:` header needed so the service can address the response back
- Mesh addressing: `service.node.amp` (e.g. `mon.mko.amp`) triggers hub mesh bridge

## AMP-addressable UI elements (Phase 1–3)

Full implementation of AMP-driven menu and widget commands, enabling external scripts and MCP tools to drive the Dioxus UI programmatically — the ARexx-at-mesh-scale vision.

### Phase 1: Menu commands
- `menu.list` — returns all menu items (id, label, shortcut, enabled, parent menu) as JSON
- `menu.invoke {"id": "preview"}` — highlights the menu item briefly, then fires the action through the same `on_action` callback as a real click
- `menu.highlight {"id": "save", "ms": 1000}` — opens parent dropdown, pulses the item
- `menu.close` — dismiss any open dropdown
- Added `MenuCommand` enum, `MENU_CMD` / `MENU_DEF` global signals, CSS pulse animation
- Integrated into `use_hub_handler` — all apps get `menu.*` for free

### Phase 2: UiRegistry + AmpButton
- `UiRegistry` — per-app registry of addressable UI elements, global signal
- `AmpButton` — drop-in component that auto-registers on mount, deregisters on unmount, reacts to incoming `UiCommand`
- `ui.list` — discover registered elements (id, kind, label, disabled)
- `ui.invoke {"id": "file.open"}` — click a button via AMP
- `ui.highlight {"id": "file.open", "ms": 500}` — visual pulse
- `ui.set {"id": "...", "value": "..."}` — set value (for future inputs/toggles)
- cosmix-view welcome screen uses `AmpButton` as proof of concept

### Phase 3: ui.batch
- `ui.batch` — process multiple `menu.*` and `ui.*` actions in a single round trip
- Returns per-action results with RC codes
- Designed for latency-sensitive cross-node scripting

### Cross-app demo
- `menu.invoke {"id": "preview"}` on cosmix-edit → highlights menu item → fires `view.open` to cosmix-view → view renders the markdown
- Fixed: `edit.open` now sets `EDITOR_PATH` immediately (not just via poll loop), so AMP-driven actions can read the current file in the same render cycle

### Architecture decisions
- AMP header layers: `id` = transport (WebSocket multiplexing), `to`/`from` = application (endpoint addressing) — like TCP sequence numbers vs IP addresses
- Security: WireGuard /24 mesh = trust domain, no per-widget ACLs needed. Bridge is the enforcement point for inter-mesh peering.
- Widget addressing: `<widget-id>.<app>.<node>.amp` — fixed 4-segment convention, widget ID is a flat semantic string
- `from:` field already on every AMP message — bridge can filter on origin without retrofitting
