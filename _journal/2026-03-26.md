# 2026-03-26 — cosmix-view: GFM markdown, DOT graph, and image viewer

## What was done

### cosmix-view (new crate)

Built a Dioxus 0.7 desktop viewer for markdown, DOT graphs, and images:

- **GFM markdown rendering** via pulldown-cmark with full GFM spec (tables,
  strikethrough, task lists, autolinks, footnotes, definition lists, smart
  punctuation). Proper monospace code blocks — the original motivation, since
  Okular renders markdown without monospace for fenced blocks.
- **DOT graph rendering** via layout-rs (pure Rust, no Graphviz dependency).
  Parses DOT format, lays out nodes, renders SVG inline. Pan/zoom/reset
  controls via JS (scroll=zoom, drag=pan, double-click=reset).
- **Image viewer** with same pan/zoom controls. Images loaded via base64 data
  URIs (WebKitGTK blocks file:// URLs; custom protocols didn't work reliably).
- **Inline DOT diagrams** in markdown: ` ```dot ` fenced blocks render as SVG.
- **Inline images** in markdown resolved from relative paths via data URIs.
- **File > Open menu** (Ctrl+O) with rfd async file picker. Keyboard shortcuts
  handled via Dioxus events (muda accelerators cause GTK warnings on Linux).
- **No-arg startup** — opens welcome screen, load files via menu.
- **CLI**: `cosmix-view [file]` or `cosmix-view -h`.

### cosmix-mail fixes

- **HeadStyles isolation**: extracted `document::Style` and `document::Link`
  into a non-reactive `HeadStyles` component to eliminate repeated Dioxus
  "Changing the props of Style {} is not supported" warnings.
- **Mailbox selection highlight**: added explicit `background:transparent` to
  unselected state — WebKitGTK was retaining the blue background when Dioxus
  updated styles without an explicit clear.

### Network topology doc

- Created DOT rendition (`_doc/2026-03-26-network-topology.dot`) of the
  existing ASCII art network topology diagram.
- Removed right-side box borders from ASCII art version — WebKitGTK's monospace
  font renders box-drawing characters at inconsistent widths.

## Decisions

- **layout-rs over Mermaid**: pure Rust, no npm/node dependency. Supports a
  subset of DOT (no cluster borders, no rank=same, limited shapes) but
  sufficient for network topology diagrams. Mermaid would require either
  npm CLI or injecting JS into WebKit.
- **base64 data URIs for images**: custom protocol (`localfile://`) via
  wry/dioxus-desktop didn't work on WebKitGTK. Data URIs work reliably
  everywhere. The 33% memory overhead is negligible for a local viewer.
- **No muda accelerators on GTK**: they cause "no accelerator installed"
  warnings. Shortcuts displayed in menu text via `\t` notation, handled
  via Dioxus keyboard events instead.

### Phase 1: appmesh foundation

#### cosmix-ui (new shared library)

Extracted duplicated code from cosmix-mail and cosmix-view into a shared crate:
- `icons.rs` — consolidated Lucide SVG icons (were duplicated across 4 files)
- `theme.rs` — dark color palette constants, button style presets
- `markdown.rs` — GFM renderer with optional DOT and image support
- `canvas.rs` — reusable pan/zoom/reset JS and CSS
- `util.rs` — file type detection, MIME helpers
- `desktop.rs` — Linux WebKit init, menu builder, file picker, shortcuts

Both cosmix-mail and cosmix-view refactored to depend on cosmix-ui.

#### cosmix-hub (new binary)

Local WebSocket message broker — the ARexx port manager equivalent:
- Axum WebSocket server on `ws://localhost:4200/ws`
- Apps connect and register with a service name (`hub.register`)
- Hub routes AMP messages between apps by `to` header
- Internal commands: `hub.register`, `hub.list`, `hub.ping`
- Responses properly echo `id` and set `type: response` for request/response matching
- ~220 lines

#### cosmix-client (new library)

App-side AMP WebSocket client:
- `HubClient::connect()` / `connect_default()` — connect and register
- `call()` — request/response with automatic id matching via oneshot channels
- `send()` — fire-and-forget messages
- `incoming()` — channel for commands from other services
- `list_services()` — convenience wrapper for `hub.list`
- ~230 lines

**Integration tested:** two clients (service-a, service-b) connected to hub,
verified service discovery (`hub.list`), message routing (A→B), and
request/response (`hub.ping`). All tests pass.
