## cosmix-view bug fix: VIEW_PATH not set from File > Open

**Root cause:** Two code paths for opening files in view:
- AMP-driven (`view.open`, welcome button) — updated `VIEW_PATH` global ✓
- Menu-driven (File > Open dialog) — only updated local `file_path` signal ✗

When scripts ran "Edit this file", `$CURRENT_FILE` was empty because `VIEW_PATH` was `None`.

**Fix:** Added `*VIEW_PATH.write() = Some(path_str)` to the file picker closure. Also added optional `path` arg to `view.show-markdown` so the "Live Markdown Preview" script can pass the source file path along with content.

## Architecture documents

### Remote Mix execution (`_doc/2026-03-31-remote-mix-execution.md`)

Vision: every cosmix node can receive a Mix script over AMP, execute it locally with access to that node's services, and return the result. The script travels to the data instead of the data traveling to the script. This is what ARexx did on the Amiga in 1990 — extended to a WireGuard mesh.

Key concept: `mix.eval` AMP command — send a script to a remote node, it runs there with access to local services (maild, mond, etc.), returns a summary. One round-trip per node instead of N queries per node.

Use cases: mesh-wide monitoring, distributed search, rolling deployment, interactive remote debugging, self-healing automation.

### Daemon consolidation (`_doc/2026-03-31-daemon-consolidation.md`)

Proposal: consolidate hub + configd + mond + logd + new mixd into a single `cosmix-noded` binary. Keep maild, indexd, webd, mcp separate (genuine reasons for isolation).

Benefits: one binary to deploy, one systemd unit, in-process hub routing, atomic startup/shutdown. The lightweight daemons (configd: 278 lines, mond: 198, logd: 77) are basically tokio tasks pretending to be processes.

### Mix-cosmix AMP enhancements (`_doc/2026-03-31-mix-cosmix-amp-enhancements.md`)

Plan to drop TOML scripting, use Mix exclusively. Key items:
- Named args in address blocks (parser change in mix-core)
- Cosmix prelude (`~/.config/cosmix/prelude.mx`) with `ensure_running()`, `wait_for_port()`
- Convert 4 TOML scripts to .mx files
- Remove TOML executor from cosmix-lib-script

## Decision: hub does NOT auto-start apps (2026-03-31)

Scripts use `ensure_running()` or `menu.launch` pattern. Keeps the hub dumb — routes messages, doesn't manage processes. Fits the ARexx model: the script is the orchestrator, not the infrastructure.

## cosmix-noded daemon consolidation

Consolidated 4 standalone daemons (cosmix-hubd, cosmix-configd, cosmix-mond, cosmix-logd) into a single `cosmix-noded` binary with 4 async modules.

**Architecture:** Option A — hub runs axum WebSocket server, client modules connect via HubClient over WebSocket within the same process. No NodeModule trait — each module has a different shape.

**Crate structure:**
- `main.rs` — CLI, spawn hub + modules, shutdown handling
- `hub.rs` — WebSocket broker (416→ same lines, from cosmix-hubd)
- `config.rs` — Config service with file watcher (from cosmix-configd)
- `monitor.rs` — System metrics on-demand (from cosmix-mond)
- `logger.rs` — AMP traffic tap writer (from cosmix-logd)

**Startup:** Hub binds listener → sends oneshot ready signal → config/monitor/logger connect as clients → `select!` on shutdown signal or hub failure.

**CLI:** `cosmix-noded [serve] --port --node --mesh-config --no-monitor --no-log`

**Tested:** `hub.list` returns `["mon","log","config"]`, `mon.status` returns system metrics, `config.get` returns settings, `amp.log` captures traffic.

**Standalone crates deleted** — cosmix-hubd, cosmix-configd, cosmix-mond, cosmix-logd removed from workspace.

**Result:** 25 crates (was 28), 3 fewer processes on a mesh node.
